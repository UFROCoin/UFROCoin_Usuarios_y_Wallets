## 1. Contexto

UFROCoin es un sistema blockchain educativo que permite:

- Registrar usuarios con wallet digital.
- Emitir monedas.
- Transferir UFROCoins.
- Consultar saldo e historial.
- Minar bloques mediante un mecanismo simplificado de **Proof of Work**.

El sistema se organiza por módulos y se integra mediante una **API REST**. Además, el proyecto exige justificar decisiones de arquitectura mediante **ADRs**.

A nivel técnico, el proyecto requiere como mínimo:

- Registro y autenticación de usuarios.
- Generación y consulta de wallets.
- Manejo de blockchain, bloques y transacciones.
- API REST para integrar módulos.
- Interfaz de usuario.
- Documentación de endpoints.
- Trabajo colaborativo con control de versiones y automatización básica.

---

## 2. Decisión

Se adopta el siguiente stack tecnológico como base del proyecto:

- **Backend / API:** Python (v3.14) + FastAPI (v0.135.2)
- **Base de datos:** MongoDB (v8.0)
- **Frontend:** Vue.js (v3.5.x) + Vite (v6.0.x)
- **Hashing / Crypto:** `hashlib` de Python
- **Autenticación:** JWT con biblioteca compatible en Python
- **SCM:** Git + GitHub
- **CI/CD:** GitHub Actions
- **Gestión del proyecto:** GitHub Projects (kanban)
- **Documentación API:** Swagger UI provisto por FastAPI

---

## 3. Justificación

### 3.1. Python (v3.14) + FastAPI (v0.135.2)

Se elige **Python + FastAPI** porque:

- Permite construir endpoints REST de forma rápida y clara.
- Integra validación de datos mediante modelos.
- Genera automáticamente documentación OpenAPI/Swagger, lo que reduce trabajo adicional.
- La versión 3.14 de Python introduce optimizaciones de memoria y tiempos de inicio más rápidos. 
- La última versión estable de FastAPI asegura el máximo rendimiento en el manejo asíncrono y compatibilidad nativa con las recientes versiones de Pydantic.

### 3.2. MongoDB (v8.0)

Se elige **MongoDB** porque:

- El dominio del proyecto trabaja naturalmente con estructuras tipo documento, como bloques, transacciones, mempool, wallets e historiales.
- Evita rigidez excesiva de esquema en una etapa donde el modelo puede evolucionar durante los sprints.
- Permite almacenar documentos JSON-like de forma muy alineada con los datos intercambiados por la API REST.
- Simplifica persistir bloques completos con su lista de transacciones embebidas.
- Utilizar la versión 8.0 garantiza contar con las últimas mejoras de seguridad, rendimiento en consultas de agregación y eficiencia en la gestión del almacenamiento.

### 3.3. Vue.js (v3.5.x) + Vite (v6.0.x)

Se elige **Vue.js + Vite** para el frontend porque:

- **Vue.js (v3.5.x):** Proporciona un entorno reactivo altamente eficiente a través de su *Composition API*, lo que resulta ideal para construir de forma modular la interfaz del explorador de la blockchain y la wallet de los usuarios. La versión 3.5 trae optimizaciones internas en reactividad y renderizado.
- **Vite (v6.0.x):** Es un empaquetador de nueva generación que ofrece un entorno de desarrollo extremadamente rápido gracias al reemplazo de módulos en caliente (HMR) instantáneo. Reduce significativamente los tiempos de compilación para producción frente a herramientas clásicas como Webpack.

### 3.4. Hashlib

Se elige **Hashlib** porque:

- Usa **SHA-256**, estándar alineado con el modelo blockchain (bloques, wallets, integridad), coherente con la guía técnica del proyecto.
- Es una librería estándar de Python, sin dependencias externas, lo que simplifica el setup, reduce errores de integración y mejora la mantenibilidad.
- Garantiza determinismo, algo crítico para validar bloques y asegurar la consistencia de la cadena (**mismo input → mismo hash**).
- Aporta simplicidad y claridad de implementación, facilitando el aprendizaje, el debugging y las pruebas unitarias en un contexto académico.

### 3.5. JWT

Se elige **JWT** porque:

- El backlog define explícitamente autenticación con token JWT y protección de endpoints.
- Es apropiado para una arquitectura frontend + API desacoplada.
- Permite transportar identidad del usuario y wallet en cada request protegida.

### 3.6. Git + GitHub

Se elige **Git + GitHub** porque:

- Centraliza control de versiones, ramas, pull requests y revisión de código.
- Se integra de forma nativa con GitHub Actions y GitHub Projects.
- Favorece el trabajo colaborativo por módulos y sprints.

### 3.7. GitHub Actions

Se elige **GitHub Actions** porque:

- Permite automatizar pruebas, linting y validaciones en cada push o pull request.
- Reduce fricción al estar integrado con el repositorio.
- Es suficiente para el alcance académico del proyecto.

### 3.8. GitHub Projects

Se elige **GitHub Projects** porque:

- Permite organizar backlog, sprint backlog, estados y responsables en formato kanban.
- Mantiene la gestión cerca del código, issues y pull requests.

### 3.9. Swagger UI

Se adopta **Swagger UI** porque:

- El proyecto requiere documentación técnica y APIs claras entre módulos.
- FastAPI la genera automáticamente, reduciendo esfuerzo manual y asegurando consistencia con los endpoints implementados.

## 4. Alternativas consideradas

### 4.1. Opción 1: Node.js + Express

**Ventaja:** ecosistema amplio y buen acoplamiento con frontend JavaScript.  
**Desventaja:** menos alineado con la guía técnica del proyecto y mayor esfuerzo para replicar ejemplos base en Python.

### 4.2. Opción 2: SQLite o PostgreSQL

**Ventaja:** mejor consistencia relacional y consultas estructuradas.  
**Desventaja:** menor flexibilidad para representar documentos complejos de blockchain durante etapas tempranas del proyecto.

---

## 5. Decisiones derivadas

A partir de este ADR, se establecen las siguientes líneas base:

- La API se diseñará bajo enfoque **REST** y será el punto principal de integración.
- Los bloques, transacciones y wallets tendrán representación orientada a documento en **MongoDB**.
- El desarrollo del lado del cliente será una Single Page Application construida con **Vue.js** y empaquetada mediante **Vite**.
- La autenticación de usuarios se hará con **JWT**.
- La documentación viva de endpoints será **Swagger / OpenAPI**.
- El pipeline mínimo de CI ejecutará validaciones automáticas sobre cada cambio al repositorio.

---

## 6. Revisión futura

Este ADR podrá revisarse si ocurre alguna de estas situaciones:

- Problemas de rendimiento o consistencia con MongoDB.
- Necesidad de migrar a una base relacional.
- Exigencias del equipo docente o integración con infraestructura distinta.
- Quiebres de compatibilidad significativos con las futuras actualizaciones del ecosistema Vue/Vite o Python/FastAPI.
