## 1. Contexto

UFROCoin es una criptomoneda educativa que será desarrollada por estudiantes durante un semestre aplicando Scrum. Nos encontramos en el Sprint 0 (Kickoff y planificación), etapa donde se deben definir los ADR, el backlog inicial y la configuración del repositorio.

El sistema se divide en 4 módulos independientes, y cada uno de los 4 equipos es responsable de un módulo. La descripción inicial sugiere que la comunicación entre módulos se realice mediante una API REST con JSON. 

Sin embargo, las operaciones core del sistema, específicamente la minería y el Proof of Work en el Módulo 3, requieren calcular un hash SHA-256 buscando un nonce que comience con N ceros. Este proceso es computacionalmente intensivo y asíncrono por naturaleza. Mantener conexiones HTTP síncronas abiertas esperando que un bloque sea minado generaría cuellos de botella y *timeouts* en el API Gateway. 

Además, existen restricciones estrictas: todo el software debe ser open source, gratuito y capaz de correr en una máquina local.

---

## 2. Decisión

Se decide adoptar desde el inicio una **Arquitectura de Microservicios** acoplada mediante **Comunicación Orientada a Eventos (EDA)**, utilizando el siguiente stack:

* **Arquitectura Base:** Microservicios (1 microservicio autónomo por cada uno de los 4 módulos asignados a los equipos).
* **Message Broker (Intermediario de Eventos):** RabbitMQ (cumple con ser open source y gratuitos).
* **Comunicación Externa (Síncrona):** API REST expuesta únicamente por el API Gateway (Módulo 4) hacia la interfaz de usuario.
* **Comunicación Interna (Asíncrona):** Patrón de publicación y suscripción de eventos (Pub/Sub) a través del broker para la interacción entre los Módulos 1, 2 y 3.

---

## 3. Justificación

### 3.1. Arquitectura de Microservicios
* Se alinea de forma nativa con la estructura organizativa del curso, donde hay 4 equipos distintos con 3 a 5 integrantes. 
* Permite que cada equipo tenga autonomía para desarrollar, estimar y probar su módulo, minimizando conflictos de código al integrar en la revisión final del sprint.

### 3.2. Comunicación Orientada a Eventos (RabbitMQ / Kafka)
* **Naturaleza asíncrona del dominio:** La solicitud de minar un bloque pendiente (Módulo 3) toma un tiempo impredecible basado en la dificultad. Usar eventos permite que el Módulo de Blockchain (Módulo 2) publique un evento de transacciones pendientes y continúe operando, reaccionando solo cuando el Módulo 3 publique el evento de bloque minado exitosamente.
* **Cumplimiento de restricciones:** RabbitMQ es una herramienta de código abierto que no requiere licencias de pago y puede ejecutarse fácilmente en contenedores dentro de una máquina local.

## 5. Decisiones derivadas

[cite_start]A partir de este ADR, el equipo asume los siguientes compromisos técnicos para el Sprint 0:

1.  [cite_start]**Contrato de Eventos:** Además de acordar los endpoints mínimos REST para el Gateway [cite: 98, 99][cite_start], los Scrum Masters en su reunión de integración [cite: 95] deberán definir el esquema JSON de los eventos asíncronos (ej. `TransactionCreated`, `BlockMined`).
2.  [cite_start]**Entorno Local Único:** Se debe configurar un archivo de orquestación (ej. `docker-compose.yml`) que incluya el broker de mensajería para asegurar que el sistema completo corra en la máquina local de cualquier estudiante[cite: 65].
3.  [cite_start]**Manejo de UI Asíncrona:** El Equipo D (Módulo 4) deberá diseñar el dashboard de manera que consulte periódicamente (polling) o escuche eventos para actualizar el estado del bloque minado[cite: 58].

## 6. Revisión futura

Este ADR se reevaluará si ocurre alguna de las siguientes situaciones:

* [cite_start]La curva de aprendizaje de RabbitMQ genera un impedimento insalvable que pone en riesgo la entrega funcional del Sprint 1 (Registro, wallet y bloque génesis).
* [cite_start]Las máquinas locales de los desarrolladores no cuentan con la memoria/CPU suficiente para ejecutar simultáneamente 4 microservicios, bases de datos y el intermediario de mensajes[cite: 65].
* [cite_start]El docente y los Scrum Masters determinan que la carga técnica está opacando el aprendizaje de la gestión con Scrum[cite: 11].
